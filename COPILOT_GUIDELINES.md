# Copilot 工作规范

## 核心原则

### 1. 禁止创建文档文件
- **规则**：不要创建任何 `.md` 文档、总结或说明文件
- **例外**：除非明确要求创建项目文档
- **原因**：保持代码库整洁，信息通过对话交流
- **违反后果**：删除所有非必要的 md 文件

### 2. 重大改动需预先通知
- **规则**：根据用户的语气判断是否需要确认
  - **纯疑问/问题**（如"为什么..."、"怎么..."、"是不是..."）→ 先回答和分析，**等待用户确认**后再改
  - **疑问 + 要求**（如"为什么...改成..."、"为什么...删除..."）→ 先简要分析，**直接执行改动**
  - **明确要求**（如"删除..."、"改成..."、"可以"、"开始"）→ 直接执行改动
  - **特大改动**（预计 30+ 分钟或涉及 3+ 文件）→ **必须先通知**，不管什么语气

- **触发条件（需要提前通知）**：
  - 涉及 3 个以上文件的改动
  - 预计耗时 30 分钟以上

### 3. 禁止擅自改变开发功能方向
- **规则**：不许随意删除或重构功能，特别是在探索替代方案时
- **具体规范**：
  - 用户问"有没有...方法"、"怎么...不用..."时 → **只提出建议和对比方案**，不执行改动
  - 建议方案时必须让用户**明确确认**（如"你要改成这个吗？"）
  - 获得用户同意后再执行：删除依赖、改动代码结构、改变实现方式
  - **禁止直接**：`npm uninstall`、删除功能、改变架构方向
- **例外**：修复 bug、解决编译错误、小范围的格式调整不受此限制
  - 改变现有功能的实现逻辑
  - 重构代码结构或架构
  - **所有 Java 文件改动**（即使只涉及 1-2 个文件）

- **执行流程**：
  1. 分析用户语气
  2. 纯疑问 → 先回答分析，等待确认
  3. 疑问+要求 或 明确要求 → 直接执行（小/中改动）或先通知（大改动）
  4. 特大改动 → 必须先通知确认
  5. 不要未经许可就开始大规模修改

## 工作流程

### 3. 代码注释和日志语言
- **规则**：所有代码注释、日志消息、错误提示必须用英文
- **范围**：适用于所有新增代码（TypeScript、Java、Kotlin 等）
- **包括**：
  - 函数/类的注释
  - 代码行注释
  - Log.d() / Log.e() 等日志消息
  - 错误提示字符串
  - 异常消息
- **不包括**：代码变量名、函数名等（保持已有命名约定）

### 3.1 Console.log 格式规范
- **规则**：所有 console.log 消息必须在前面标注出处，格式为 `[FileName]`
- **目的**：便于追踪日志来源，方便调试和问题排查
- **格式示例**：
  ```javascript
  // ✅ 正确做法
  console.log("[webhookService] Order received: ", orderId);
  console.log("[tokenManager] Token expired, fetching new one");
  console.log("[localService] Saving order to storage");
  
  // ❌ 错误做法（无出处标注）
  console.log("Order received: ", orderId);
  console.log("Token fetching...");
  ```
- **适用于**：
  - 业务逻辑日志（正常操作）
  - 调试日志
  - 错误和警告消息
- **命名规范**：
  - 使用文件名或模块名：`[webhookService]`、`[tokenManager]`、`[localService]`
  - 对于路由文件：`[posServiceRoutes]`、`[webhookRoutes]`
  - 对于工具类：`[tokenManager]`、`[uberService]`

### 4. API 地址必须有文档支持
- **规则**：禁止臆想 API 端点地址，所有 API 调用必须有官方文档或已验证的实现支持
- **执行方法**：
  - 使用前先检查 `apiDoc/` 文件夹中的 Swagger/OpenAPI 文档
  - 不确定的 API 地址必须询问用户或查阅官方文档
  - 如果 API 文档不存在，必须标注为 TODO 并通知用户
- **违规例子（禁止）**：
  - ❌ `https://api.uber.com/v2/eats/v2/stores` （虚构的）
  - ❌ 自己编造的端点路径
- **正确做法**：
  - ✅ 查证实际 Swagger 文档：`/v1/delivery/stores` （正确）
  - ✅ 标注文档来源：`// From: Uber Eats Marketplace Store API`
